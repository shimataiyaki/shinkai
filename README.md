# Shinkai

「組織の可観測性」を高めるための、実験的プロダクトシリーズ。

---

## 1. なぜ Shinkai なのか

組織の停滞は、個人の怠惰ではありません。**「声を上げても応答がない」「誰が何を知っているか見えない」** という、可観測性の欠如が生み出す、**合理的な沈黙**です。

Shinkai は、「気がつく人」が疲弊し、チームが壊れる前に、情報の流れを可視化し、**組織を停滞から「学習する組織」へ変える**ための実験基盤です。

---

## 2. 解決する問題と理想状態

| 問題 | 理想状態 |
| :--- | :--- |
| 情報が属人化し、誰が何を知っているか見えない | 全員が必要な情報にアクセスできる |
| 発言しても応答がなく、行動が無駄になる | 行動が記録され、進捗として意味を持つ |
| チーフが能動的に「確認しに行く」疲弊 | 普段の場で、状態が受動的に届く |
| プロジェクトが「やりっぱなし」で終わる | 経験が教訓として継承される |

---

## 3. 全体アーキテクチャ

### ユーザー視点の2層構造

Shinkai は **Web（設計室）** と **アプリ（実行部隊）** の2層で構成されます。

```
┌─────────────────────────────────────────────┐
│                  Shinkai                     │
└─────────────────────────────────────────────┘
        │
        ├── Web（計画・設計・評価）
        │   ├── Vision Planner（ビジョン定義）
        │   ├── Conductor（権限設計）
        │   ├── Dashboard（状態可視化）
        │   └── Reflector（評価・学習）
        │
        └── アプリ（日常・実行・応答）
            ├── Connector for LINE（タスク抽出・状態応答）
            └── Connector for Voice（口頭記録・将来）
```

### 見えない背骨：統合レイヤー

両者をつなぐ基盤。ユーザーからは隠蔽されています。

-   **Shinkai ID** (`proj-xxxx`) による全データの統合
-   **共通データストア**（スプレッドシート）
-   **状態応答エンジン**（チーフの問い合わせに返答）
-   **テキストマイニング**（Geminiによる評価分析）

### ゲーム理論的設計

Shinkai は、組織の「ゲームのルール」を書き換えます。

| 機能 | ゲーム理論的効果 |
| :--- | :--- |
| 発言の自動抽出 | 協調のリターンを上げる |
| 進捗・停滞の可視化 | フリーライド（沈黙）のコストを上げる |
| 及第点の明示 | 「やっても無駄」のリスクを下げる |

---

## 4. プロダクト群

### Web（設計室）

-   **Vision Planner**: プロジェクトの「なぜ」と及第点を定義。定例会議の管理。Shinkai IDを発行。
-   **Conductor**: タスクに権限を付与。「誰が何をどこまで決めていいか」を可視化。
-   **Dashboard**: 進捗、タスク一覧、チームの状態を表示。Shinkai IDでログイン。
-   **Reflector**: プロジェクト終了時に三軸評価（健全性・納得感・学び）を実施。自由記述からAIが教訓を抽出。

### アプリ（実行部隊）

-   **Shinkai Connector for LINE** (旧 e-colleague): LINE上の会話からタスクを自動抽出・記録。「状況」と聞けば状態を返信する。
-   **Shinkai Connector for Voice** (構想): 口頭ミーティングを記録・文字起こしし、タスクを抽出する。

---

## 5. データのライフサイクル

```
[計画] → [実行] → [応答] → [評価] → [学習] → [次の計画へ]
```

1.  **計画**: Vision Planner でプロジェクトを定義
2.  **実行**: Connector が日々の会話からタスクを抽出・記録
3.  **応答**: チーフの「状況」に統合レイヤーが進捗と停滞を返す
4.  **評価**: 及第点達成時に Reflector で振り返り
5.  **学習**: Gemini がテキストマイニングで教訓を抽出
6.  **継承**: 次のプロジェクトの Vision Planner で過去の教訓を参照

---

## 6. 技術スタック

無料枠内での動作を前提とします。

-   **Web**: GitHub Pages
-   **バックエンド**: Google Apps Script (GAS)
-   **データストア**: Google スプレッドシート
-   **AI**: Gemini API (無料枠)
-   **チャット**: LINE Messaging API (Replyのみ、無料枠)

---

## 7. 開発ロードマップ

1.  **Phase 1 (MVP)**: 統合レイヤー基盤 & Connector for LINE の「状態応答」機能
2.  **Phase 2**: Vision Planner (Web) のプロトタイプ & Shinkai ID 連携
3.  **Phase 3**: Dashboard (Web) のプロトタイプ & Reflector による評価機能
4.  **Phase 4**: Conductor (権限設計) の実装 & テキストマイニング機能の強化
5.  **Phase 5**: Connector for Voice の検証 & マルチプロジェクト管理

---

## 8. Shinkai の及第点

**チーフがWebでプロジェクトを立ち上げ、LINEにConnectorを招待する。チームは普段通り会話し、タスクは自動で記録される。「状況」と聞けば進捗が返ってくる。プロジェクトが終わればWebで振り返り、教訓が次のプロジェクトに継承される。**
